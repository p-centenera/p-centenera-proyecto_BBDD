GUIA ORDENADA PARA ENTENDER Y CUSTOMIZAR EL SITIO DJANGO
========================================================

1) OBJETIVO DE ESTA GUIA
------------------------
Esta guia sirve para:
- Entender la estructura actual del proyecto Django.
- Identificar donde se conecta la base de datos.
- Ver donde estan las consultas SQL y la logica de negocio.
- Definir como mantener y configurar correctamente MySQL.
- Definir como reemplazar consultas actuales por consultas nuevas.


2) ESTRUCTURA ACTUAL DEL PROYECTO
---------------------------------
Raiz del proyecto:
- manage.py
  Punto de entrada para ejecutar comandos Django (runserver, migrate, etc.).

- miproyecto_gbd/
  Paquete principal con configuracion y logica del backend.

- templates/
  HTML de las paginas.

- static/css/main.css
  Estilos visuales.

- README.md
  Documentacion basica.

Carpeta miproyecto_gbd/:
- settings.py
  Configuracion global Django (incluye DATABASES, INSTALLED_APPS, etc.).

- urls.py
  Enrutado principal de URLs.

- views.py
  Controladores de peticiones HTTP (conectan formularios, consultas y plantillas).

- forms.py
  Formularios Django usados por las vistas.

- sqlUtil.py
  Utilidades SQL: funciones para ejecutar o construir consultas.

- funciones.py
  Logica auxiliar del proyecto.

- asgi.py / wsgi.py
  Configuracion de despliegue.

Carpeta templates/:
- base.html
  Plantilla base comun.

- navbar.html
  Navegacion comun.

- home.html, formulario.html, euribor.html, euriborMes.html
  Vistas HTML concretas de la app.


3) FLUJO FUNCIONAL (RESUMEN)
----------------------------
1. El usuario abre una URL definida en urls.py.
2. Django llama a una funcion en views.py.
3. La vista procesa el formulario (forms.py) y/o llama a ejecutar la logica (funciones.py, sqlUtil.py).
4. Se ejecutan consultas contra la base de datos.
5. La vista devuelve una plantilla de templates/ con resultados.


4) CONFIGURACION MYSQL (FIJA EN ESTE PROYECTO)
-----------------------------------------------
Paso A: Mantener dependencias necesarias
- Instalar/actualizar los paquetes Python requeridos para MySQL.
- Dependencias usadas en este proyecto: mysqlclient, mysql-connector-python, pandas, matplotlib.

Paso B: Mantener variables de entorno
- Definir DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD.
- Evitar credenciales hardcodeadas en el codigo.

Paso C: Verificar coherencia de configuracion
- settings.py debe usar django.db.backends.mysql.
- sqlUtil.py debe conectarse a la misma base MySQL que settings.py.

Paso D: Verificar esquema esperado
- La tabla euribor debe existir con columnas compatibles:
  anio, mes, valor.

Paso E: Verificacion
- Probar conexiones.
- Probar formularios y vistas clave.
- Confirmar que templates muestran los resultados correctamente.


5) COMO CAMBIAR LAS CONSULTAS ACTUALES POR OTRAS
-------------------------------------------------
Paso 1: Inventario de consultas
- Localizar todas las consultas en sqlUtil.py y, si aplica, en views.py/funciones.py.
- Registrar para cada consulta:
  - Nombre de funcion
  - Entrada esperada (parametros)
  - Salida esperada (columnas, tipo de dato)
  - Pantalla/plantilla que depende de esa salida

Paso 2: Definir nuevas necesidades
- Especificar claramente:
  - Que datos se quieren consultar
  - Filtros
  - Agrupaciones
  - Orden
  - Limites
  - Formato de salida

Paso 3: Reescribir capa de acceso a datos
- Implementar nuevas funciones en sqlUtil.py con nombres claros.
- Mantener una interfaz estable para minimizar cambios en views.py.

Paso 4: Ajustar vistas y formularios
- Actualizar views.py para llamar a las nuevas funciones SQL.
- Ajustar forms.py si cambian campos de entrada.

Paso 5: Ajustar templates
- Actualizar las plantillas que renderizan los resultados para reflejar nuevas columnas o estructura.

Paso 6: Pruebas
- Probar caso nominal y casos borde.
- Verificar errores de conversion de tipos y valores nulos.


6) PROPUESTA DE REFACTORIZACION MINIMA (RECOMENDADA)
-----------------------------------------------------
Para facilitar mantenimiento en MySQL y cambios de consultas:
- Centralizar todo el SQL en sqlUtil.py (o modulo datos/ dedicado).
- Evitar SQL incrustado directamente en views.py.
- Crear funciones pequenas y orientadas a un caso de uso.
- Estandarizar formato de salida de funciones (lista de dict, tuplas, etc.).
- Documentar cada funcion SQL con:
  - Proposito
  - Parametros
  - Salida
  - Dependencias de motor SQL

8) RIESGOS COMUNES Y PREVENCION
-------------------------------
- Riesgo: Desalineacion entre settings.py y sqlUtil.py en la conexion MySQL.
  Prevencion: usar las mismas variables de entorno en ambos puntos.

- Riesgo: Cambios de columnas rompen templates.
  Prevencion: validar estructura de salida y actualizar HTML al mismo tiempo.

- Riesgo: Credenciales hardcodeadas.
  Prevencion: usar variables de entorno y archivo .env.

- Riesgo: Regresiones por cambios en consultas.
  Prevencion: pruebas por cada vista clave con datos reales de ejemplo.


9) SIGUIENTE PASO PRACTICO SUGERIDO
-----------------------------------
Empezar por:

2. Listar funciones SQL actuales en sqlUtil.py.
3. Crear un mapa "consulta actual -> consulta nueva".
4. Implementar y probar una sola vista completa de punta a punta.

10) APLICADO A ESTE PROYECTO (MAPA REAL)
----------------------------------------
Situacion actual detectada:
- settings.py usa MySQL en DATABASES['default'].
- sqlUtil.py abre conexion MySQL manual con mysql.connector.
- funciones.py consulta directamente la tabla euribor usando SQL en texto.
- forms.py y views.py dependen de esas consultas para poblar ano/mes y resultados.

Conclusión importante:
- El proyecto queda fijado a MySQL como unica base de datos.
- Se mantiene SQL manual centralizado en sqlUtil.py y funciones.py.

Inventario real de consultas actuales:
- consultaEuribor(mes, anio):
  select valor from euribor where anio=... and mes=...

- anios():
  select distinct anio, anio from euribor order by anio

- meses(anio):
  select distinct mes, mes from euribor where anio=... order by mes

- graficoEuribor() y graficoEuriborMes():
  select mes, anio, valor from euribor

Contrato de datos actual (lo que esperan forms/views/templates):
- Tabla principal: euribor
- Columnas minimas esperadas:
  - anio (entero)
  - mes (entero)
  - valor (numerico)
- anios() y meses() deben devolver pares (valor, etiqueta) para ChoiceField.

Como mantener MySQL sin romper la app:

Paso 1: Parametrizar conexion en sqlUtil.py
- Reemplazar host/database/user/password hardcodeados por variables de entorno:
  DB_HOST, DB_NAME, DB_USER, DB_PASSWORD, DB_PORT

Paso 2: Alinear settings.py
- Mantener DATABASES en MySQL y consistente con las mismas credenciales.
- Documentar que las consultas principales se ejecutan mediante sqlUtil.py.

Paso 3: Validar esquema en MySQL
- Crear la tabla equivalente a euribor con columnas compatibles.
- Cargar datos historicos antes de probar formularios y graficos.

Paso 4: Ejecutar pruebas funcionales criticas
- Home.
- Formulario: carga de anos, carga dinamica de meses, consulta de valor.
- Graficos: euribor.png y euriborMes.png generados en static/.

Como cambiar a OTRAS CONSULTAS DIFERENTES (ejemplo de metodo):

Paso A: Definir nuevo caso de uso
- Ejemplo: consultar promedio anual en lugar de valor mes puntual.

Paso B: Crear nueva funcion en funciones.py
- promedioEuriborAnual(anio)

Paso C: Crear/ajustar SQL en sqlUtil.py
- select avg(valor) from euribor where anio=...

Paso D: Ajustar forms.py
- Si cambia entrada (por ejemplo solo ano), modificar campos y validaciones.

Paso E: Ajustar views.py
- Llamar a la nueva funcion y pasar resultado al template.

Paso F: Ajustar template correspondiente
- Mostrar nuevo dato/tabla/grafico con los nombres de contexto actualizados.

Paso G: Refinado visual (CSS y favicon)
- Revisar la hoja de estilos principal para ajustar colores, tipografias, espaciados y jerarquia visual.
- Definir una paleta coherente (colores de fondo, texto, enlaces, botones y estados hover/active).
- Unificar estilos de navegacion, formularios, tablas y titulos para mantener consistencia.
- Validar que la interfaz se vea bien en escritorio y movil (responsive).
- Añadir favicon del sitio y referenciarlo en la plantilla base para reforzar identidad visual.
